pre-FDE bridge artifact

Lead가 사용자 맥락을 개발 가능한 SIC 재료로 번역하는 HTML

사용자는 긴 맥락을 주고, Lead는 그 맥락을 해석해 매번 이 템플릿을 전문화한다. 화면은 비개발자 언어, 개발자 산출물, 승인 전 금지 경계를 한 줄로 연결해 FDE session 전에 무엇을 합의해야 하는지 보이게 만든다.

1. User context 사용자는 문제와 배경을 자연어로 준다

완성된 요구사항이 아니어도 된다. 가치관, 사례, 우려, 원하는 판단 장면이 원재료다.

2. Lead translation Lead가 맥락을 UI와 질문으로 전문화한다

빈칸을 사용자에게 떠넘기지 않고, Lead가 먼저 해석한 뒤 사용자가 맞고 틀림을 검토한다.

3. Developer artifact SIC 후보와 DTC 보류 항목으로 넘긴다

개발자가 볼 noun, verb, evidence, action boundary와 아직 구현하면 안 되는 범위를 분리한다.

1. Lead가 사용자 context를 해석해 HTML을 전문화하기

사용자는 원문 맥락을 제공하고, Lead는 그 맥락을 domain, 운영 결정, ontology seed, non-goal로 먼저 번역한다. 사용자는 빈칸 작성자가 아니라 Lead 해석의 검토자다.

lead-specialized layer
사용자가 주는 것

긴 설명, 사례, 불편함, 목표, 걱정, 좋은 결과의 느낌. 개발 용어로 정리되어 있지 않아도 된다.

Lead가 하는 것

사전 context를 해석해 5-turn 질문, 후보 noun/verb, 근거 경계, 금지 action을 채운 뒤 HTML로 보여준다.

사용자가 검토하는 것

Lead 해석이 맞는지, 빠진 우려가 있는지, SIC 초안으로 넘겨도 되는 경계인지 확인한다.

2. 5-turn batch decision surface

각 turn은 사용자가 결정해야 할 내용을 3개 이하로 압축한다. Lead는 답변을 모아 SIC 초안의 intent, scope, noun/verb, non-goal, evidence 경계로 옮긴다.

turn card decision spec
1mission

무엇을 위해 이 ontology를 만드는가

“도구를 만들고 싶다”가 아니라 “어떤 판단을 더 잘하고 싶은가”를 분리한다.

SIC intent사용자가 얻고 싶은 운영 판단
scope이번 단계에서 다룰 대상 범위
success리뷰 가능한 성공 상태
open아직 질문해야 하는 내용
2ontology

지식그래프의 기준 단위를 어디까지 쪼갤 것인가

atomic concept까지 과하게 쪼개기보다, 사용자의 실제 판단에 필요한 최소 단위를 정한다.

nounsOntology object 후보
links관계와 의존성 후보
properties각 noun에 필요한 속성 후보

3evidence

무엇을 근거로 판단하고, 무엇은 추정으로 남길 것인가

좋은 FDE는 claim-source-gap을 먼저 분리한다. 근거가 약한 추천은 추천처럼 보이면 안 된다.

evidence주장별 근거 자료
confidence확신/가설/미정 구분
review사람 검토가 필요한 지점
privacy보관/노출 금지 경계
4action

그래프가 사용자에게 어떤 action을 제안할 수 있는가

recommendation은 강한 기능이다. “보여주기”, “검토 요청”, “자동 실행”을 분리해야 한다.

read참조 화면과 지표
suggest승인 전 추천 후보
writeDTC 이후 가능한 변경
block지금 금지할 action
5contract

SIC 초안에 넣을 것과 DTC로 넘길 것을 분리한다

SIC는 의미의 경계이고 DTC는 실제 변경의 경계다. HTML 답변은 mutation 승인이 아니라 계약 초안 재료다.

includeSIC 초안 포함 항목
excludenon-goal과 deferred scope
approve사용자 승인이 필요한 항목

3. recommendation과 risk의 근거 표기 기준

권장사항은 “무엇을 뒷받침하는가”와 “무엇을 증명하지 못하는가”를 같이 보여주어야 한다. 그래야 자동화처럼 보이는 문장을 과신하지 않는다.

source ledger
판단
근거
왜 지지하는가
증명하지 못하는 것
5-turn batch는 긴 사용자 맥락을 SIC 재료로 바꾸기에 적합하다.
FDE lead skill, SIC/DTC stage map, HITL recipe
의미 확인, scope, evidence, action, contract 경계를 단계별로 분리한다.
모든 도메인에서 최적의 turn 수라는 점은 증명하지 않는다.
HTML은 비개발자 리뷰에 유리하다.
사용자 요구, HITL HTML pattern taxonomy, task-framing-canvas recipe
긴 질문을 화면 단위 결정과 요약으로 나누어 검토할 수 있다.
HTML 자체가 workflow 승인이나 mutation 권한을 만들지는 않는다.
선택형 HTML workflow는 사용자가 요청할 때에만 실행해야 한다.
Semantic gate 결과, DTC governance, user approval boundary
사전 context 주입과 Lead specialization은 SIC 초안 재료를 만들지만 모든 FDE session의 자동 단계는 아니다.
사용자 요청 후 승인된 contract 안에서 workflow에 연결할 수 없다는 뜻은 아니다.
외부 UI/HCI best practice 근거는 보수적으로 표시해야 한다.
SOURCE_POLICY.md와 required-sources manifest의 blocked status
현 로컬 source pack이 complete direct capture 상태가 아니다.
이 UI가 쓸모없다는 뜻이 아니라, final evidence-grade claim이 아직 이르다는 뜻이다.
비개발자 bridge UI는 사용자 언어와 개발자 언어를 나란히 보여줘야 한다.
NN/g usability heuristics, Digital.gov plain language
사용자에게 익숙한 말로 먼저 설명하고, 개발자가 볼 noun/verb/action boundary로 번역해야 기억 부담이 줄어든다.
이것만으로 ontology가 정확하다는 점은 증명하지 않는다.
질문은 필요한 이유와 사용처가 보여야 한다.
GOV.UK form structure question protocol
각 질문이 왜 필요한지, 답을 어디에 쓸지, 누가 답해야 하는지 드러나야 불필요한 입력을 줄일 수 있다.
모든 질문을 한 화면에 넣어도 된다는 뜻은 아니다.
입력 UI에는 visible label과 예시가 필요하다.
W3C WCAG 2.2 SC 3.3.2
사용자가 어떤 정보를 넣어야 하는지 보이는 label과 instruction이 있어야 한다.
접근성 전체를 보장한다는 뜻은 아니며, 추가 WCAG 검증은 별도다.

4. Lead가 반드시 되물어야 하는 위험

아래 항목은 specialization 때 open decision으로 남겨야 한다. 답이 없으면 SIC에 확정 문장처럼 쓰면 안 된다.

open decisions
  • !
    사용자 가치와 시스템 기능을 혼동하는 위험

    예: “학생을 잘 지도하고 싶다”는 가치이고 “오답 유형 그래프를 본다”는 기능이다. 둘을 분리해야 SIC intent가 흔들리지 않는다.

  • !
    지식그래프 입도를 너무 작게 만드는 위험

    atomic concept 집착은 리뷰 비용을 늘릴 수 있다. 실제 판단에 필요한 중간 단위를 먼저 정해야 한다.

  • !
    추천을 action 승인처럼 보이게 만드는 위험

    검토해야 하는 제안은 read-only 또는 review-required로 표시해야 한다. 자동 실행은 DTC 이후 별도 승인 대상이다.

  • !
    근거 없는 개인화가 초개인화처럼 보이는 위험

    개인화는 관찰, 풀이, 평가 근거와 연결되어야 한다. 근거가 없으면 hypothesis로 남겨야 한다.
