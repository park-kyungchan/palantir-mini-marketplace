비개발자 관점의 한 줄 정의

palantir-mini는 Codex에게 “작업을 끝까지 책임지는 절차”를 입혀준다

Codex는 실제 파일을 읽고 고치고 테스트할 수 있는 런타임입니다. palantir-mini plugin은 그 힘에 작업 경계, hook 정책, MCP 도구, lineage 기록, release gate를 붙여서 “빠른 수정”을 “검증 가능한 운영 흐름”으로 바꿉니다.

01 · 의미

요청을 바로 구현하지 않는다

SIC/DTC가 필요한지 판단하고, 승인되지 않은 운영 의미는 추측으로 바꾸지 않습니다.

02 · 실행

Codex-native 도구로 움직인다

MCP, hook, subagent, shell, git 같은 Codex 표면을 씁니다. 다른 런타임 기능을 자동 상속한다고 말하지 않습니다.

03 · 보호

편집 전 영향도를 묻는다

PreToolUse와 pre_edit_impact가 파일 영향, DTC 정책, write scope를 확인합니다.

04 · 증거

끝난 뒤 검증을 남긴다

테스트, PR, merge SHA, CI gap, reload 필요 여부가 함께 남습니다.

사용자 프롬프트부터 끝까지

Codex 세션 안에서 벌어지는 7단계

사용자는 그냥 부탁하지만, 내부에서는 “요청-의미-경계-실행-검증-릴리스-학습”이 차례로 움직입니다.

Step 01

사용자 프롬프트가 들어온다

promptId, promptHash, sessionId가 붙어 결과가 어디서 시작했는지 추적할 수 있습니다.

Step 02

의미와 변경 경계를 확인한다

문서 산출물처럼 위험이 낮으면 not_required로 통과하고, 운영 의미 변경이면 승인 경계를 요구합니다.

Step 03

읽을 자료와 영향 범위를 좁힌다

local source, BROWSE/INDEX, ontology_context_query, 웹 리서치를 서로 다른 권위로 분리합니다.

Step 04

복잡하면 subagent에게 나눈다

Lead는 직접 구현하고, 시나리오/런타임 사실/UX 검토처럼 충돌 없는 레인을 읽기 전용 subagent에게 맡깁니다.

Step 05

파일 편집 전 hook이 정책을 적용한다

이 HTML도 생성 전 pre_edit_impact를 거쳤고, generated 파일과 plugin cache는 금지 범위로 유지했습니다.

Step 06

검증과 릴리스 증거를 붙인다

테스트 통과, CI 계정 상태 gap, PR 본문, merge commit을 서로 섞지 않고 기록합니다.

Step 07

다음 세션이 재생할 수 있게 남긴다

중요한 결정은 events, lineage, replay, self-check로 이어져 다음 Codex 세션이 이어받을 수 있습니다.

복잡한 요청이 어떤 길로 처리되는지 눌러서 보기

각 시나리오는 “비개발자가 실제로 물어볼 법한 요청”을 기준으로 만들었습니다. 버튼을 선택하면 해당 workflowFamily에서 Codex와 palantir-mini가 무엇을 확인하고 실행하는지 바뀝니다.

Codex active runtime에서 중요한 상태 신호

“무엇을 할 수 있는가”와 “무엇을 주장하면 안 되는가”

palantir-mini는 Codex의 실제 능력과 다른 런타임의 능력을 섞지 않습니다. 설치된 plugin payload, 로컬 source checkout, Codex cache, Claude/Gemini 미설치 상태를 분리해서 봅니다.

source SSoT수정해야 할 곳

/home/palantirkc/palantir-mini-marketplace/plugins/palantir-mini/가 의미 변경의 기준입니다.

runtime consumer편집하면 안 되는 곳

~/.codex/plugins/cache/**는 설치된 소비자 payload입니다. 소스처럼 패치하지 않습니다.

hooks정책은 자동 또는 수동으로 적용

Codex hook이 있으면 실행하고, native parity가 없으면 정책 의도와 gap을 명시합니다.

MCP tools도구 호출도 검증 대상

semantic gate, intent router, ontology context, pre-edit impact 같은 도구가 작업 레일을 만듭니다.

subagents분업은 범위 계약으로 시작

읽기 전용 verifier와 구현 worker를 구분하고, write scope가 겹치면 병렬화하지 않습니다.

reload문서만 바꾸면 재시작 불필요

MCP, hook, skill, manifest 변경은 Codex 재시작이 필요하지만 이 HTML 문서는 runtime reload가 필요 없습니다.
