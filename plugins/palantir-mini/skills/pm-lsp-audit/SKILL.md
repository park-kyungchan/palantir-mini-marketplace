# LSP Audit — TypeScript Code Integrity Analysis

This skill audits a TypeScript codebase using the LSP tool for precise,
compiler-grade analysis. It finds dead code, boundary violations, and type
safety issues that grep-based tools miss.

All work is Lead-Direct — no delegation to subagents for execution.

## Scope Detection

From `$ARGUMENTS`:
- `--scope home` → `~/.claude/` infrastructure (schemas, rules)
- `--scope project` → current working directory
- `--scope all` → both
- Empty → default to `--scope project`

## Phase 1: Dead Code Detection

### Step 1: Enumerate files in scope

Use Glob to find all `.ts`/`.tsx` files.

**Exclude** (do not run LSP on):
- `_generated/` — auto-generated by Convex
- `node_modules/`, `.bun/` — external dependencies
- `*.test.ts`, `*.spec.ts` — test files
- `ontology/*.ts` — declarative definitions (not executable code)
- `*.config.ts`, `*.d.ts` — config and ambient types

### Step 2: Enumerate exports per file

For each file in scope, use LSP `documentSymbol`:

```
LSP: documentSymbol(filePath, line=1, character=1)
```

Collect exported symbols: Function, Variable, Interface, Class, Constant.
Cross-check by reading the file line to verify `export` keyword presence.

### Step 3: Find references for each export

For each exported symbol:

```
LSP: findReferences(filePath, line=<symbol line>, character=<symbol name start>)
```

Filter out self-references (in the declaring file). If 0 external references:
- `type`/`interface` → **orphan type** (severity: warning)
- All other symbols → **dead export** (severity: warning)

**Skip these** (entry point exceptions):
- `default` export in `pages/`, `app/` files — route components
- Re-exports in `index.ts` — the source file is the authority
- Convex `query()`/`mutation()`/`action()` wrappers — consumed by framework
- `main`, `app`, `App` — application entry symbols

### Step 4: Check incoming calls for functions

For each Function symbol:

```
LSP: prepareCallHierarchy(filePath, line=<fn line>, character=<fn name start>)
LSP: incomingCalls(filePath, line=<fn line>, character=<fn name start>)
```

`prepareCallHierarchy` must be called before `incomingCalls`. Both use same position.

If 0 callers AND not an entry point → **unreachable function** (severity: warning).
For exported functions, cross-check with Step 3: only flag if BOTH show zero usage.

### Step 5: Check unused imports

For each file, use LSP `hover` on line 1 to trigger diagnostics:

```
LSP: hover(filePath, line=1, character=1)
```

Collect diagnostic codes:
- **6133** (`declared but value is never read`) → unused import (severity: info)
- **6196** (`declared but never used`) → unused declaration (severity: info)

## Phase 2: Dependency Chain Analysis

### Step 1: Parse import statements

For each file, extract imports via regex: `import .* from ["'](.+)["']`

### Step 2: Resolve imports via LSP

For each import:
```
LSP: goToDefinition(filePath, line=<import line>, character=<path start>)
```
Record the resolved file path.

### Step 3: Check boundary rules

These boundary rules come from `~/.claude/rules/01-ontology-first-core.md` (meaning → ontology → contracts → runtime ordering):

**Ontology domain isolation (3 rules):**

| Rule | Check | Severity |
|------|-------|----------|
| DATA → LOGIC/ACTION | File in ontology DATA domain must NOT import from LOGIC or ACTION | error |
| LOGIC → ACTION | File in ontology LOGIC domain must NOT import from ACTION | error |
| ACTION → DATA/LOGIC | Allowed (consumers import producers) | — |

**Cross-layer isolation (2 rules):**

| Rule | Check | Severity |
|------|-------|----------|
| convex/ → src/ | convex/ must NOT import from `../src/` | error |
| src/ → convex/ | src/ may only import from `convex/_generated/` (codegen types) | error |

**Frontend layer boundaries (3 rules):**

| Rule | Check | Severity |
|------|-------|----------|
| hooks → api only | `features/*/hooks/` must import from `convex/_generated/api` only | error |
| components → hooks | `features/*/components/` imports from `../hooks/` + `src/components/` | error |
| shared → no entity | `src/components/` must NOT import entity-specific modules | error |

### Step 4: Detect circular imports

Build module dependency graph from resolved imports. Run DFS cycle detection.
Circular import → severity: error.

## Phase 3: Type Safety Hardening

### Check 1: Explicit `any` usage

Grep for `any` type annotations: `/:\s*any\b/`, `/as\s+any\b/`
Severity: info.

### Check 2: Loose assertion (`as` cast)

Grep for `\bas\b` keyword in `.ts` files.
For each cast, use LSP `hover` on the expression to check if the cast is necessary.
Severity: info.

### Check 3: Missing branded types

Cross-reference ontology value type definitions from `ontology/data.ts`.
Use LSP `hover` on property access sites to check for raw `string`/`number`
where branded types are expected.
Severity: info.

## Phase 4: Interaction Schema Validation

When the project has gesture-interactive components (useDrag, usePinch, useGesture,
or touch-action CSS), run cross-file checks from `~/.claude/schemas/interaction/`.

These 7 HC constants need LSP/code-level analysis that declaration-based validators
(`schemas/interaction/validator.ts`) cannot perform:

### Step 1: Gesture handler isolation (HC-INT-05, HC-INT-06, HC-ELEM-02)

```
Grep: useDrag|usePinch|useGesture|onTap|onClick  →  find all gesture-owning components
LSP: findReferences(gestureHookImport)  →  locate all gesture handler registrations
```

For each component with gesture handlers:
- **HC-INT-05**: Does the component have a `data-entity-*` or entity mapping declaration?
  Cross-ref against ontology `data.ts` entity list. Missing = orphan UI.
- **HC-INT-06**: Does any child of this gesture-owning component also have `onClick`,
  `onTap`, `onPointerDown`, or gesture hook bindings? If yes = dual-handler violation.
- **HC-ELEM-02**: Is a gesture-isolated component (own `useDrag`/`usePinch`) nested
  inside a gesture-owning parent? Check JSX nesting via AST/LSP.

### Step 2: Mutation idempotency (HC-BIND-05)

For each Convex mutation that can be triggered by gestures:
```
Grep: throw.*ConvexError  →  in mutation handlers triggered by gesture bindings
```
If a mutation `throw`s on bounds/duplicate instead of returning gracefully,
rapid gesture input (4 swipes < 100ms) causes unhandled error noise.
Correct: `return { currentValue, atEnd: true }` instead of throw.

### Step 3: Async state initialization (HC-BIND-06)

```
Grep: useReducer.*\?\?.*0|useReducer.*\.length\s*\?\?\s*0
```
State machines initialized with `useQuery() ?? 0` or `data?.length ?? 0`
capture stale defaults before async data arrives. The reducer initializes
once on mount — Convex/SWR data arrives ~200ms later but the state is locked.

### Step 4: Swipe detection pattern (HC-GEST-04)

```
Grep: swipe\[0\]|swipe\[1\]|velocity.*threshold  →  velocity-based swipe detection
```
@use-gesture's built-in `swipe[]` uses velocity at pointer release. On laptop
touchscreens (Galaxy Book, Surface), users decelerate before lifting — release
velocity drops below threshold even for fast swipes. Correct: displacement +
time window, not velocity at release.

### Step 5: Touch target size (HC-ELEM-05)

Requires runtime measurement (Playwright or computed styles). Flag components
with `gesture-target` role where rendered dimensions < 24×24 CSS px.
Note: this check is best combined with `/runtime-audit` for runtime verification.

## Fix Plan

After generating the audit report, prioritize and execute fixes:

**Wave 1 — Boundary violations** (errors): Fix import paths, move misplaced code.
**Wave 2 — Dead code removal** (warnings): Remove unused exports, orphan types.
**Wave 3 — Type safety** (info): Add branded types, remove unnecessary casts.
**Wave 4 — Verification**: Re-run `/lsp-audit` to confirm zero errors.

Apply fixes sequentially. Verify with typecheck (`bun run tsc --noEmit`) at
each wave boundary before proceeding.

## Gotchas

- **LSP server not running → empty results**: If the LSP tool returns no symbols or diagnostics, the TypeScript language server may not be initialized. Verify the project has a `tsconfig.json` in scope and call `LSP: hover` on a known symbol to confirm the server is alive before running the full scan. _Why: a silent empty result looks like a clean codebase when it's actually a dead session._

- **Dynamic imports are falsely flagged as dead code**: Symbols consumed via `import(...)` or `require(...)` strings have zero static references, so `findReferences` returns 0 even though the code is active. Cross-check Phase 1 findings against grep for the symbol name before marking dead. _Why: tree-shaking-friendly patterns and plugin registries use dynamic loading by design._

- **Re-exports trigger false boundary violations**: A barrel `index.ts` that re-exports from multiple domains will look like a DATA→LOGIC import when it is just a passthrough. Only flag the originating import, not the re-export chain. _Why: re-exports have no runtime coupling — the consumer's actual import target is what matters for boundary enforcement._

- **Missing `node_modules` causes unresolved types**: If `bun install` has not been run, `goToDefinition` on external package types returns `null` and type-safety checks produce spurious errors. Confirm `node_modules/` exists before Phase 3. _Why: absent dependencies make every third-party type appear as `any`, flooding the report with noise._

- **Circular import detection scope is file-level only**: The DFS cycle detector in Phase 2 Step 4 operates on resolved file paths. Circular re-exports within the same directory (sibling `index.ts` re-importing the parent) may not form a graph cycle at the file level but still cause runtime issues. Flag these separately with a manual check. _Why: the LSP graph cannot see runtime module evaluation order, only static import edges._
