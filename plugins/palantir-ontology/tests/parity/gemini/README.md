# Gemini-Side Parity Fixture (A640 seed for A650)

Ledger row A640 (docs/architecture.md ADR-007, execution-plan DoD item 9:
"Claude, Codex, and Gemini consume identical semantic fixtures. Packaging
differences are adapter metadata only."). Mirrors
`tests/parity/codex/README.md` and `tests/parity/claude/README.md`'s exact
pattern (A620/A630).

## What this directory contains

`inventory-shape.fixture.json` — a deterministic structural snapshot of
`src/adapters/gemini/binding.generated.ts`'s generated-inventory SHAPE:

- `capabilityAreas` — the sorted 8 R210 capability-area names.
- `toolNames` — the sorted generated MCP tool names.
- `manifestFieldKeys` — the sorted top-level manifest field names (one more
  than Codex/Claude's set — see "Gemini-specific field" below).
- `toolInputSchemaConvention` — the flat-schema convention every tool's
  `inputSchema` follows (`type: "object"`, `requiredKeys: ["runtime"]`,
  `additionalProperties: false`, `flat: true`).
- `nativePackagingSupported` — `false`, fixed: this marketplace ships no
  native Gemini extension packaging convention at generation time (see
  `src/adapters/gemini/README.md`'s "Native packaging" section).

Generated by `generate-fixture.ts` (`bun run tests/parity/gemini/generate-
fixture.ts`) — pure function of the committed Gemini binding, no
hand-authored fixture content.

## What A650 should (and should not) compare

**Should be byte-equivalent across codex/claude/gemini** (this is the part
ADR-007's "generated inventories" parity requirement actually covers):
`capabilityAreas` (same 8 area names regardless of runtime),
`manifestFieldKeys`'s SHAPE convention (each runtime's own manifest fact
carries the same field *categories* — provider identity, capability
verdicts, unsupported/unknown lists), and the flat-schema convention
(`toolInputSchemaConvention`).

**Should NOT be asserted byte-equivalent**: the actual capability VERDICT
VALUES inside each runtime's own binding (Gemini's `hooks` verdict is
`supported`; a different runtime's equivalent area may legitimately differ
per R210) — those are runtime-specific facts, not semantic decisions, and
R210 itself records them as differing per runtime ("Packaging differences
are adapter metadata only" — metadata is allowed to differ; semantics is
not). Do not build a parity assertion that would fail merely because Codex
and Gemini have different official capability facts.

## Gemini-specific field: `nativePackagingSupported` / `manifest.nativePackaging`

Unlike Codex (`.codex-plugin/plugin.json`) and Claude
(`.claude-plugin/plugin.json`), this marketplace has no native Gemini
extension packaging convention built yet, so `GEMINI_BINDING.manifest`
carries one field Codex/Claude's manifests do not:
`nativePackaging: { supported: false, transportMode: "neutral-mcp-cli",
note }`. This means `manifestFieldKeys` will legitimately NOT be
byte-identical to Codex/Claude's own `manifestFieldKeys` (9 keys here vs. 8
there) — per ADR-007 DoD item 9's own text, "packaging differences are
adapter metadata only," this is exactly that: Gemini-specific packaging
metadata, not a semantic-decision field and not evidence of a
hand-maintained fork. `A650` should treat this as a documented, allowed
per-runtime difference (the field's PRESENCE/ABSENCE is the metadata
difference the DoD line anticipates), not attempt to force byte-identical
`manifestFieldKeys` across all three runtimes.

## Scope note

A640 adds the Gemini side — the last of the 3 Wave-6 runtime rows
(`src/adapters/{codex,claude}/` already existed from A620/A630). This
directory is the Gemini-side input `A650` (same writer, same tree,
`depends_on [A640]`) consumes, alongside `tests/parity/codex/` and
`tests/parity/claude/`'s existing inputs, not a self-contained 3-way parity
proof. `fixture.test.ts` in this directory only proves the Gemini-side
fixture itself is well-formed and traceable to the committed binding.
